## A. 変更ファイルの棚卸し

まず、今回のセッションで変更したファイルを確認する。

1. **個人プロジェクト**: `git status` で未コミットの変更を確認する
2. **共有プロジェクト**（CLAUDE.mdの「共有プロジェクト」にパスが記載されている場合のみ）:
   - `git -C （共有リポジトリのパス） status` で未コミットの変更を確認する
   - `git -C （共有リポジトリのパス） pull` を実行し、他メンバーの変更がないか確認する
   - 他メンバーの変更があったら「〇〇ファイルが他のメンバーに更新されています」とユーザーに報告する
   - コンフリクトが発生したらCLAUDE.mdの手順に従いユーザーに確認する

変更したファイルの一覧（個人・共有の両方）をユーザーに提示する。

## B. 記憶の更新

1. **短期記憶の更新**: `memory/YYYY-MM-DD.md`（今日の日付）に今回の作業内容・途中経過・引き継ぎ事項を追記する
   - そのファイルが既にあれば末尾に `---` 区切りで追記
   - なければ新規作成
   - **共有プロジェクトで作業した内容も、個人プロジェクトの短期記憶に記録する**

2. **長期記憶の確認**: 以下に該当するものがあれば記録する

   **振り分けルール（書き先の判断）**:
   記録すべき内容が見つかったら、まず「どこに書くか」を判断する：

   | 内容の性質 | 書き先 | 例 |
   |---|---|---|
   | 特定施策の方針・戦術 | その施策のドキュメント | アウトバウンド施策.md、コンテンツ戦略.md 等 |
   | スキルの実行ルール | そのスキルのSKILL.md | 診断レポートのフィルタ基準 等 |
   | ツールの手順・仕様 | `docs/*.md` | HubSpot連携手順 等 |
   | 領域横断 or 上記に該当しない | `memory/PERSISTENT.md` | Claude Codeの制約、汎用的な実務Tips 等 |

   **判断の基準**: 「この情報は、該当するドキュメントやスキルに書けば十分か？」をまず考える。ドキュメントに書ける内容をPERSISTENT.mdに重複して書かない。該当するドキュメントがまだ存在しない場合は新規作成する（ユーザーに確認を取ってから）。

   **「学び」と「方針」を区別する**:
   - **学び**: 実際にやってみて分かった、状況が変わっても使える実務知見（例: 「紙ではハイパーリンクが機能しない→URL直接記載」）→ 記録する
   - **方針・仮説**: 「こうやってみよう」と決めたこと（例: 「二面攻めでアプローチする」）→ 施策ドキュメントに書く。PERSISTENT.mdに「学び」として固定しない（将来の選択肢を狭めるため）

   **記録すべき内容**:
   - **ユーザー指摘からの学び**: 作業中にユーザーから「こうしてほしい」「次からはこうして」等の指摘・修正フィードバックを受けた場合、同じ指摘を繰り返さないよう記録
   - **施策結果からの改善**: 実施した施策の結果（成功・失敗・数値）が判明した場合、次回に活かせる知見を記録
   - **確定した方針・意思決定**: 今後の判断基準として再利用できるもの

## C. 変更の波及チェック

今回変更したファイルの内容が、他のファイルに影響しないか確認してください：

1. **変更ファイルの特定**: このセッションの会話履歴から、自分が変更・作成したファイルを洗い出す（`git diff` は並行セッションの変更が混ざるため使わない）
2. **依存関係の洗い出し**: 変更したファイルを参照・前提としている他のファイルがないか、Grep/Globで探す
3. **整合性チェック**: 依存ファイルの内容が、今回の変更と矛盾していないか確認
4. **対応**:
   - 矛盾・不整合があれば → ユーザーに報告し、確認を取ってから修正
   - 影響なし or memory/設定ファイルのみの変更 → 「波及影響：なし」と報告

例: ターゲット定義を変更 → キーワード戦略・コンテンツ計画・営業資料なども更新が必要かもしれない

## D. ファイル整理の検討

今回のセッションで変更・作成したファイルについて、以下の観点でレビューしてください：

### ファイル名
- 内容を正確に表しているか？曖昧・汎用的すぎないか？
- プロジェクトの命名規則（日本語、バージョンはvN、日付NG）に沿っているか？

### ファイル分割
- 1ファイルに複数の関心事が混在していないか？
- ファイルが大きくなりすぎていないか？（目安：300行超は分割検討）
- 逆に、細かく分けすぎて見通しが悪くなっていないか？

### フォルダ構成
- ファイルが適切なフォルダに置かれているか？
- 同じフォルダ内にファイルが増えすぎていないか？（目安：10ファイル超はサブフォルダ検討）
- 新しいサブフォルダを作って整理した方が良いケースはないか？

**判断**: 問題なければ「ファイル整理：問題なし」と報告。改善提案があればユーザーに提案して確認を取ってから実行してください。

## F. コミット・プッシュ

基本方針：**コミットしたら必ずプッシュまで実行する**（個人・共有とも）。途中状態を残さず、リモートに反映するところまでが1作業単位。

1. **個人プロジェクト**:
   - 変更したファイルをすべてコミットする
   - `git push` を実行してリモートに反映する
   - push が rejected された場合は `git pull --rebase` してから再度 push する
   - コンフリクトが発生したらCLAUDE.mdの手順に従いユーザーに確認する
2. **共有プロジェクト**（Step Aで未コミット変更があった場合のみ）:
   - `git -C （共有リポジトリのパス） add` → `commit` → `push` を実行する
   - push が rejected された場合は `git pull` してから再度 push する
   - コンフリクトが発生したらCLAUDE.mdの手順に従いユーザーに確認する

**例外**: ユーザーが明示的に「pushしないで」「コミットだけ」と指示した場合のみpushをスキップする。
